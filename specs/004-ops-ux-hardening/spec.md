## 用户场景与测试（必填）

### 用户故事 1 - 移动端窄屏可正常使用（优先级：P1）

用户主力设备是 Android 手机。打开线上站点登录后，登录页与主仪表盘（待办列表、新增代办、日历、随记收件箱等）在窄屏下应排版正确、内容不溢出、按钮可点、表单可填写，能够完成"登录 → 查看待办 → 新增一条待办 → 勾选完成"的核心闭环。

**优先级原因**：用户最高频的使用场景就是手机端；窄屏排版若错乱，等于核心体验不可用，价值最高。

**独立测试**：在 ≤420px 宽的视口下打开站点，逐屏检查登录页与仪表盘无横向滚动、无元素重叠、关键操作可完成，即可单独验证并交付"手机可用"的价值。

**验收场景**：

1. **假设** 在 390px 宽视口打开站点，**当** 进入登录页，**那么** wordmark、说明文字、邮箱/密码输入框、登录/注册按钮均完整可见、不溢出、可操作。
2. **假设** 已登录且窄屏，**当** 查看主仪表盘，**那么** 待办区与侧栏面板自动堆叠为单列，日历保持 7 列周历且不撑破容器，无横向滚动条。
3. **假设** 窄屏下，**当** 新增一条待办并勾选完成，**那么** 全流程可顺利操作，复选框、下拉、日期选择均可点击命中。

---

### 用户故事 2 - 版本可见且能收到更新提示（优先级：P1）

用户和维护者需要知道线上当前运行的是哪一版，并在有新版本部署后，使用中的页面能提示"发现新版本 → 更新页面"，避免一直停留在被缓存的旧界面（此前正是因为缓存导致看到的是旧样式）。

**优先级原因**：更新提示链路当前失灵，用户拿不到新前端；同时版本可见是排查问题的基础。直接影响"改了能不能让用户看到"。

**独立测试**：部署一版后查看页面版本徽标是否显示真实构建标识（非 `UNKNOWN`）；再部署新版后，已打开的旧页面是否出现更新提示。

**验收场景**：

1. **假设** 线上已部署，**当** 用户查看顶部版本徽标，**那么** 显示可识别的真实版本标识，而非 `UNKNOWN`。
2. **假设** 用户已打开页面，**当** 后台部署了一个新版本，**那么** 页面在合理时间内出现"发现新版本"提示，点击后可加载到最新界面。

---

### 用户故事 3 - 清库/换库后用户不掉登录（优先级：P2）

运维在清空或更换数据库后，已登录用户的会话不应失效。系统允许把签名密钥固定为一个由部署方控制的稳定值，使其独立于数据库内容，从而在数据库重置后仍能验证既有登录凭证。

**优先级原因**：保障会话稳定性，避免运维动作误伤用户登录态；属于可靠性兜底，重要但触发频率低于前两项。

**独立测试**：在配置了稳定签名密钥的前提下，清空/重建数据库后，使用此前签发的凭证仍能保持登录；未配置时维持现有行为。

**验收场景**：

1. **假设** 部署方提供了稳定的签名密钥，**当** 数据库被清空或更换，**那么** 此前签发且未过期的登录凭证仍然有效，用户无需重新登录。
2. **假设** 部署方未提供稳定签名密钥，**当** 系统启动，**那么** 行为与当前一致（自动生成并持久化于数据库），不产生回归。

---

### 用户故事 4 - 提交即受自动化质量门禁保护（优先级：P3）

维护者每次提交 Pull Request 或向主分支推送时，系统应自动运行既有测试，并校验后端可正常启动（导入），把明显的回归挡在合并之前。

**优先级原因**：面向维护者的质量基线，不直接面向终端用户，但能显著降低后续迭代引入回归的风险；作为长期资产价值高、紧迫性较低。

**独立测试**：提交一个会让测试失败的改动，确认自动化检查给出失败结论并阻止"绿色合并"；提交一个正常改动则通过。

**验收场景**：

1. **假设** 打开一个 PR，**当** 自动化检查运行，**那么** 既有测试与后端导入校验全部执行，结果（通过/失败）清晰可见。
2. **假设** PR 中包含会导致测试失败的改动，**当** 检查运行，**那么** 检查结论为失败，作为合并前的明确信号。

### 边界场景

- 维护者本地仍以 SQLite 方式运行时，版本标识在没有任何外部注入的情况下应回退到一个合理值（如本地 git 提交或 `unknown`），不得报错。
- 同时提供了稳定签名密钥与数据库中已存在的密钥时，必须有明确且稳定的优先级，避免两者不一致导致随机掉登录。
- 极窄屏（≈320px）下，过滤器、日历、弹窗等密集组件不得出现横向滚动或控件不可点。
- 自动化检查在缺少可选外部依赖（如 AI、飞书相关）时，应能跳过或不因此误判失败。
- 本特性不得创建、修改或删除任何用户的待办、子任务、账号或同步数据。

## 需求（必填）

### 功能需求

- **FR-001**：登录页与主仪表盘必须在窄屏（手机宽度）下正确排版：无横向滚动、无元素重叠、关键控件可点击命中。
- **FR-002**：窄屏下多列布局（待办区与侧栏、表单网格、日历）必须自适应折叠，且日历保持周/月的列对齐不被撑破。
- **FR-003**：用户必须能在窄屏下完成核心闭环：登录、浏览待办、新增待办、勾选完成。
- **FR-004**：系统必须在页面上展示可识别的真实版本标识；在部署环境下不得显示为 `UNKNOWN`。
- **FR-005**：当线上部署了新版本时，已打开的页面必须能向用户提示存在更新，并提供更新到最新界面的操作。
- **FR-006**：系统必须支持由部署方配置一个稳定的签名密钥，使会话验证独立于数据库内容；该稳定值在存在时优先级高于数据库中保存的密钥。
- **FR-007**：未配置稳定签名密钥时，系统必须维持现有行为（自动生成并在数据库中持久化），不得产生回归或掉登录。
- **FR-008**：每次 Pull Request 与向主分支推送时，系统必须自动运行既有测试并校验后端可正常导入，并清晰反馈通过/失败结论。
- **FR-009**：自动化检查在缺少可选外部依赖时必须能稳健处理（跳过相关用例），不因环境差异误报失败。
- **FR-010**：本特性不得改变任何现有 API 的行为或契约，也不得改动既有业务逻辑。
- **FR-011**：除非本规格明确包含迁移或删除流程，否则系统必须保护现有待办、子任务、账号、同步和配置数据。
- **FR-012**：相关部署与配置说明（版本标识来源、稳定签名密钥配置）必须以中文补充进文档。

### 关键实体（如功能涉及数据则填写）

- **版本标识**：代表线上当前运行的构建版本，用于页面展示与"是否有更新"的判断依据；来源可由部署环境提供，本地有合理回退。
- **签名密钥**：用于校验登录凭证的密钥；可由部署方以稳定值提供，或由系统自动生成并持久化，二者有明确优先级。

## 成功标准（必填）

### 可衡量结果

- **SC-001**：在 390px 与 320px 两个窄屏宽度下，登录页与仪表盘均无横向滚动、无重叠，核心闭环（登录→新增→勾选完成）100% 可完成。
- **SC-002**：线上页面版本徽标显示真实版本标识的比例为 100%（不再出现 `UNKNOWN`）。
- **SC-003**：部署新版本后，已打开的旧页面能够获得更新提示并成功加载到新界面（可在一次验证中复现）。
- **SC-004**：在配置稳定签名密钥的前提下，数据库被清空/更换后，既有未过期登录凭证仍 100% 保持有效。
- **SC-005**：故意引入会失败的测试改动时，自动化检查 100% 给出失败结论；正常改动则通过。

## 假设

- 移动端目标主要为现代移动浏览器，参考宽度取 320–420px；不针对超小或异形屏做特别适配。
- 版本标识在部署环境可由该环境提供构建/提交信息；本地开发回退到 git 提交或 `unknown`。
- 稳定签名密钥由部署方在环境层面配置，属于运维动作，不在应用内提供管理界面。
- 自动化检查复用仓库 `tests/` 下既有测试，不在本特性内新增大规模测试套件。
- 现有数据库结构、认证机制与 API 契约保持不变，本特性仅做加固与体验修复。
