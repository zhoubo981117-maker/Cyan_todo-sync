# 研究结论：飞书随记入口与 AI 收件箱统一整理

## 决策 1：飞书入口复用 records 流程，而不是直接创建 todos

**Decision**：飞书有效文本消息统一创建 `record`，再执行现有 record AI 整理流程；正式任务只由 Web/PWA 中的草稿确认动作创建。  
**Rationale**：这与上一轮“随记收件箱 + AI 整理”的人工确认原则一致，可以避免飞书随手输入直接污染正式任务列表。  
**Alternatives considered**：继续沿用现有 `create_feishu_todo` / `create_feishu_ai_todos` 直接建任务路径；该方案速度快，但与本轮规格冲突，且难以保留原始记录审查链路。

## 决策 2：飞书消息归属默认账号，同时记录发送者信息

**Decision**：飞书消息仍写入 `TODO_FEISHU_DEFAULT_EMAIL` 对应账号，但 record 需要保存飞书发送者信息。  
**Rationale**：当前系统已有默认账号配置，能最快落地；保留发送者信息可以支持后续用户绑定、迁移或审计。  
**Alternatives considered**：要求飞书用户先绑定账号；该方案长期更严谨，但会显著扩大本轮范围，并阻断当前个人/小团队入口。

## 决策 3：用飞书消息事件 ID 做幂等去重

**Decision**：新增或复用 record 来源字段保存飞书消息事件 ID，并以该 ID 识别重复送达。  
**Rationale**：飞书平台重试通常携带稳定消息 ID，用它去重比内容相似度窗口更准确，也避免误合并用户主动重复发送的内容。  
**Alternatives considered**：按发送者 + 文本 + 时间窗口去重；该方案可处理缺失 ID 的情况，但容易误伤真实重复记录。本轮按规格只要求事件 ID。

## 决策 4：飞书成功和失败都回复

**Decision**：飞书处理完成后，成功和失败都回复状态；成功回复仅说明已保存/已整理，不提供飞书内确认任务入口。  
**Rationale**：用户在聊天入口需要即时反馈确认消息没有丢；同时保持任务确认必须回到 Web/PWA，避免扩大交互范围。  
**Alternatives considered**：只失败回复或不回复；干扰更少，但用户不容易判断随手记是否进入系统。

## 决策 5：不新增依赖和后台队列

**Decision**：本轮继续使用当前 Python 进程内处理路径；AI 慢或失败时保存 record 并返回失败/处理中状态，不引入队列服务。  
**Rationale**：项目宪法要求简洁可观测，当前规模是个人/小团队自托管；引入队列会增加部署复杂度。  
**Alternatives considered**：新增后台任务队列或独立 worker；可靠性更强，但对当前范围过重。
