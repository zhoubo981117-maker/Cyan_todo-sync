# Todo Sync 项目宪法

## 核心原则

### I. 个人数据归属
用户的待办、子任务、账号状态、重置令牌、AI 输入和同步元数据，必须保留在项目明确控制的存储或已配置的外部服务中。默认存储模型是 `TODO_DATA_DIR` 指向的本地 SQLite。任何功能在迁移、同步、部署和本地开发过程中，都必须保护已有数据。

任何会删除、重写、导出用户数据，或把用户数据发送到新的外部系统的改动，都必须在功能规格中明确说明，在实施计划中记录，并提供回滚或恢复路径。

### II. 离线友好的同步
本项目的核心目标，是让 Android、Windows、Web 和可选 Flutter 客户端通过同一个服务账号保持一致。任何涉及待办或子任务行为的改动，都必须保护稳定的 client ID、tombstone、`updated_at` 语义，以及 `/api/sync/pull` 和 `/api/sync/push` 合约。

每个可独立交付的功能，都必须说明它在刷新、重连、离线同步追赶、重复任务、已完成任务和已删除记录场景下的行为。同步相关改动必须保持向后兼容；如果无法兼容，规格必须包含迁移计划和用户可理解的升级说明。

### III. 可审查的 AI
AI 功能必须把供应商密钥保留在后端，禁止把密钥暴露给浏览器、PWA、Flutter 或飞书客户端。AI 输出在进入持久化之前，必须经过解析、校验、长度限制和清洗。

AI 默认必须生成可审查的草稿或计划；只有在规格明确说明的可信自动化路径中，才允许自动创建真实任务，例如已配置默认账号的飞书入口。依赖日期或时间的提示词，必须包含当前 Asia/Shanghai 日期或时间；模型返回无效时，必须提供确定性的降级逻辑或受控错误。

### IV. 测试门禁
认证、密码重置、AI 整理、飞书接入、同步、数据迁移和重复任务等行为变更，必须包含自动化测试，才可以视为完成。窄范围 UI 改动可以使用人工浏览器验证，但计划中必须说明为什么自动化覆盖不实际。

测试必须避免真实第三方网络调用，AI、SMTP 和飞书客户端都应使用 mock 或等价替身。验证必须包含最小相关测试命令；共享后端改动还必须包含 `python -m py_compile server.py`。

### V. 简洁且可观测的实现
本仓库优先保持小型自托管技术栈：Python 标准库服务端、SQLite、原生 Web 资源、可选 Flutter 客户端、Docker 和 Caddy 部署，以及显式环境变量。新增依赖、框架、后台服务或数据存储，必须有明确的用户价值，并记录被拒绝的更简单替代方案。

重要运行行为必须提供确定性的 JSON 错误、不会泄露密钥的服务端日志，以及在适用时提供简单的健康检查或版本验证路径。

## 平台、安全与文档约束

后端目标运行环境是 Python 3.11+，本地开发以 Windows 为主，VPS 部署以 Linux 为主。Web 客户端必须继续作为浏览器应用和 PWA 使用；除非规格明确证明必要，否则不得引入前端构建流程。Flutter 客户端是离线优先的补充入口，不能成为唯一支持的客户端路径。

密钥必须通过环境变量或被版本控制排除的本地数据文件配置。密码必须使用加盐 PBKDF2 哈希，或使用经过评审的更强替代方案。令牌和重置流程必须有时效限制，并且不得泄露账号是否存在，除非这是已在代码或规格中说明的有意 UX 取舍。

临时本地或 VPS 验证可以使用 HTTP/IP 访问。公开生产式部署应该使用 Caddy 或等价反向代理提供 HTTPS。CORS 改动必须限定在实际部署需要内，并在计划中记录。

所有项目文档、规格、计划、任务拆解、变更记录、归档说明和面向用户的说明，默认必须使用中文。代码标识符、命令、环境变量、API 路径、错误码和第三方专有名词可以保留英文。若必须使用英文文档，必须在规格或计划中说明原因。

## 开发流程

每个功能在实现前，都必须先在 `specs/` 下形成面向用户的规格，紧急生产修复除外。规格必须定义可独立测试的用户故事、边界场景、数据处理预期和可衡量成功标准。计划必须列出会影响的准确仓库路径，并在任务生成前通过宪法检查。

任务拆解必须保持用户故事可独立交付。后端、前端、Flutter、部署和飞书相关改动，如果触及不同文件，应该拆成不同任务。可以并行执行的任务，必须避免重叠写入同一文件或同一责任边界。

在声明工作完成之前，执行者必须运行计划中列出的验证命令；如果无法运行，必须说明原因。环境变量、部署命令或公开 API 行为发生变化时，必须同步更新生成文档和用户可见的使用说明。

所有重要结论、阶段总结和归档记录必须使用中文保存，方便后续回看。提交信息可以使用英文或中文，但面向用户和项目维护的正文记录优先中文。

## 治理

本宪法优先于冲突的本地习惯、生成的计划和 agent 建议。未来功能如果需要违反某条原则，计划中必须记录违反项、为什么当前必须这样做，以及被拒绝的更简单方案。

修订宪法必须通过明确的 Spec Kit 宪法更新完成。版本号遵循语义化版本：MAJOR 表示不兼容的治理变化，MINOR 表示新增或实质扩展原则，PATCH 表示澄清和文字修正。每次修订必须包含同步影响报告，并检查 Spec Kit 模板是否需要同步。

所有规格、计划、任务、代码评审和完成总结，都必须检查受影响范围内的宪法合规性。未解决的合规缺口会阻塞实施，除非用户明确接受已记录的风险。
