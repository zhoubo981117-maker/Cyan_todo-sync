# 任务清单：[FEATURE NAME]

> 文档语言：除代码标识符、命令、环境变量、API 路径、错误码和第三方专有名词外，本任务清单必须使用中文撰写并归档。

**输入**：来自 `/specs/[###-feature-name]/` 的设计文档  
**前置条件**：plan.md（必需）、spec.md（用户故事必需）、research.md、data-model.md、contracts/

**测试要求**：认证、密码重置、AI 整理、飞书接入、同步、数据迁移、重复任务或共享 API 合约等行为变更，必须包含自动化测试。窄范围 UI 改动可以使用人工验证，但 plan.md 必须说明为什么自动化覆盖不实际。

**组织方式**：任务按用户故事分组，确保每个故事可以独立实现、独立测试和独立演示。

## 格式：`[ID] [P?] [Story] 描述`

- **[P]**：可并行执行，前提是不同文件且没有未完成依赖。
- **[Story]**：该任务属于哪个用户故事，例如 US1、US2、US3。
- 描述中必须包含准确文件路径。

## 路径约定

- **当前仓库常见结构**：`server.py`、`web/`、`tests/`、`flutter_app/`、`deploy/`、`docs/`
- **后端行为测试**：优先放在 `tests/`
- **Web/PWA 改动**：优先放在 `web/`
- **Flutter 改动**：优先放在 `flutter_app/lib/`
- 下方任务是示例，生成真实 `tasks.md` 时必须根据 plan.md 调整。

## 依赖与执行顺序

### 阶段依赖

- **准备工作（Phase 1）**：无依赖，可以立即开始。
- **基础设施（Phase 2）**：依赖 Phase 1 完成，会阻塞所有用户故事。
- **用户故事（Phase 3+）**：都依赖 Phase 2 完成。
  - 可以在基础设施完成后并行推进。
  - 也可以按优先级 P1 -> P2 -> P3 顺序推进。
- **打磨阶段**：依赖计划范围内的用户故事完成。

### 用户故事依赖

- **用户故事 1（P1）**：基础设施完成后即可开始，不依赖其他故事。
- **用户故事 2（P2）**：基础设施完成后即可开始，可能与 US1 集成，但必须可独立测试。
- **用户故事 3（P3）**：基础设施完成后即可开始，可能与 US1/US2 集成，但必须可独立测试。

### 单个用户故事内部顺序

- 必需测试必须先写，并在实现前确认失败。
- 模型/数据结构先于服务。
- 服务先于 endpoint 或 UI 接入。
- 核心实现先于集成。
- 一个故事完成并通过检查点后，再进入下一个优先级故事。

### 可并行机会

- 所有标记 `[P]` 的准备任务可以并行。
- Phase 2 中标记 `[P]` 的基础任务可以并行。
- Phase 2 完成后，不同用户故事可以并行。
- 同一用户故事中的多个测试任务可以并行。
- 同一用户故事中互不依赖、不同文件的模型任务可以并行。

## 实施策略

### MVP 优先（只做用户故事 1）

1. 完成 Phase 1：准备工作。
2. 完成 Phase 2：基础设施。
3. 完成 Phase 3：用户故事 1。
4. 停下来验证：独立测试用户故事 1。
5. 如果可用，进行演示或部署。

### 增量交付

1. 完成准备工作和基础设施。
2. 添加用户故事 1，独立测试，演示或部署。
3. 添加用户故事 2，独立测试，演示或部署。
4. 添加用户故事 3，独立测试，演示或部署。
5. 每个故事都必须增加价值，同时不破坏之前故事。

### 并行团队策略

多人协作时：

1. 团队共同完成准备工作和基础设施。
2. 基础设施完成后：
   - 开发者 A：用户故事 1
   - 开发者 B：用户故事 2
   - 开发者 C：用户故事 3
3. 各故事独立完成并集成。

## 备注

- `[P]` 表示不同文件、无依赖，可以并行。
- `[Story]` 标签把任务映射到对应用户故事，便于追踪。
- 每个用户故事都应该能独立完成和独立测试。
- 实现前必须确认必需测试失败。
- 每个任务或逻辑任务组完成后可以提交。
- 在任何检查点都可以停下来独立验证。
- 避免：模糊任务、同文件冲突、破坏独立性的跨故事依赖。
